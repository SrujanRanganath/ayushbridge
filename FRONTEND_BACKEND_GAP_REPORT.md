Frontend ↔ Backend Compatibility Audit

Scope
- Pages audited: DashboardPage, SearchPage, MappingPage, BundleViewerPage, LoginPage, RegisterPage
- Services audited: api.ts, searchService.ts, mappingService.ts, fhirService.ts, dashboardService.ts

Findings (by page)

1) SearchPage ↔ GET /api/v1/search
- Frontend expects array of objects with { code, name, category, description, score }
- Backend `search` route returns array of documents with selected fields: code, name, category, description, and adds `score` via textScore. Match: OK
- Notes: Frontend uses `api` instance which injects JWT; route is protected — ensure user is authenticated before calling from UI. SearchPage uses protected route correctly (app applies protect to /search). No mismatch.

2) MappingPage ↔ POST /api/v1/map
- Frontend `mappingService.mapNAMASTECode` posts to `http://localhost:5000/api/v1/map` and expects `MappingResult` with fields: icdCode, icdTitle, confidence, source, conditionName?, namasteCode?
- Backend `/map` returns on cache or new result: { namasteCode, icd11Code, icd11Title, confidence, source }
- MISMATCH: Field names differ — frontend expects `icdCode`/`icdTitle` while backend returns `icd11Code`/`icd11Title`. This causes runtime undefined fields in UI.
- Current code working? The MappingPage references `mappingResult.icdCode` and `mappingResult.icd11Code` in different places (inconsistency). This is a bug.
- Recommendation: Align frontend `MappingResult` shape with backend (`icd11Code`/`icd11Title`) or normalize service response.

3) Bundle generation ↔ POST /api/v1/fhir/bundle
- Frontend `fhirService.generateFHIRBundle` posts to `http://localhost:5000/api/v1/fhir/bundle` and expects a FHIR Bundle JSON in response.
- Backend `/fhir/bundle` constructs a Bundle and returns it. Match: OK
- Note: frontend stores returned bundle in router state and shows JSON — shape is dynamic but acceptable.

4) DashboardPage ↔ GET /api/v1/dashboard
- Frontend `dashboardService.fetchDashboard` expects `DashboardResponse { recentMappings, recentBundles }`.
- Backend `/dashboard` returns { mappings: { data, total, page, pages }, bundles: { data, total, page, pages } }
- MISMATCH: Frontend expects simple arrays `recentMappings` and `recentBundles`, while server returns wrapped objects `mappings` and `bundles` with paging metadata.
- Effect: UI may break; in current commit I replaced mock UI and used `data.recentMappings` leading to undefined. However earlier build succeeded because services and UI were wired to axios and did not strictly reference missing properties at compile time. At runtime, the UI will not display expected data.
- Recommendation: Update frontend `dashboardService.fetchDashboard` to adapt to backend response shape or update server to return expected shape. Prefer adding a stable API contract (OpenAPI).

5) Auth flow
- Frontend `api` attaches token to requests from `localStorage` and handles 401 by clearing token and redirecting to `/login`.
- Backend `protect` middleware requires `Authorization: Bearer <token>` header and validates JWT.
- Both sides compatible: OK

6) General inconsistencies
- Some frontend services use centralized `api` (which applies baseURL and auth), others use direct full axios calls to `http://localhost:5000` (mappingService, fhirService). This bypasses centralized config and may cause CORS/auth inconsistencies.
- Recommendation: Switch all services to use `api` instance.

7) Mock/dummy data
- Several UI pages previously contained mock/hardcoded statistics (DashboardPage earlier had hardcoded stats) — our rule forbids adding dummy data. Recent modifications introduced components that expect `DashboardResponse.recentMappings`/`recentBundles` (also a mismatch). Ensure UI mirrors actual server response.

Summary
- Critical mismatches: /map response field names, /dashboard response shape, inconsistent axios usage across services.
- Non-critical but important: absence of FHIR validation, mapping confidence calibration, and missing RBAC.
